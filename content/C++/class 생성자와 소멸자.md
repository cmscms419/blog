# class 생성자와 소멸자

## 1. new를 사용한 생성자 호출

동적할당을 할 때, 사용된다. 

C에서는 malloc, calloc, realloc 같은 함수를 이용해서 heap 메모리에 저장공간이 생기고 주소값을 반환해 준다.

이 역할을 C++에서는 new, delete 연산자를 사용한다.

- new는 위 3가지와 같은 역할을 한다
- delete는 free함수와 같이 동적할당을 제거해주는 역할을 한다.

+ new를 했는데 메모리에 공간이 없으면 NULL을 반환한다.

## 2. 생성자의 오버로딩

생성자는 클래스를 선언하게 되면, 그 클래스는 메모리에 객체로써 자리를 잡게 된다. 그 객체를 초기화하기 위해서 생성자를 통해서 초기화 해 줄 수 있다.

생성자 특징

- 생성자의 이름은 항상 클래스명과 똑같아야 한다.
- 리턴값이 없다

생성자 또한 함수의 일종으로 오버로딩이 가능하다. 기존의 함수에 인자값이 다르게 해서 생성자를 만들 수 있다.

## 3. 소멸자의 오버로딩

결론적으로 소멸자 오버로딩은 불가능하다.

소멸자는 컴파일러에 의해서 자동으로 호출되는데, 소멸자라는 문맥을 사용자가 지정할 수 없다.

→ 파라미터를 지정할 수 없으므로, 오버로딩을 할 수 없다.

## 4. 묵시적(implict) 변환

생성자 파라미터가 1개인 경우, 생성자를 호출하기 위해 대입문을 사용할 수 있다. 하지만 대입문으로 번역되는 것이 아니라, 컴파일러가 생성자 호출로 변환시킨다. 이것을 **묵시적 변환**이라고 한다.

```cpp
// 예를 들어
class Ctest
{
public:
	Ctest(int i);
	~Ctest()
	{

	}

private:

};

int main()
{
	Ctest t(3);

	return 0;
}
```

## 5. 생성자와 소멸자 직접 호출 불가능한가

일반적인 것은 아니다. 하지만 명시적(explicitly) 호출은 가능하다.

### 생성자

임시 개체를 만들기 위해서 생성할 수 있다.

```cpp
// 예를 들어
class Ctest
{
public:
	Ctest(int i);
	~Ctest()
	{

	}

private:

};

int main()
{
	Ctest t(3);

	t = Ctest(2);
	return 0;
}
```

이런 식으로 **임시 개체**를 만들 수 있다. 다만 이것은 메모리 적으로 낭비이다.

### 소멸자

```cpp
// 예를 들어
class Ctest
{
public:
	Ctest(int i);
	~Ctest()
	{

	}

private:

};

int main()
{
	Ctest t(3);

	t.~Ctest();
	return 0;
}
```

이렇게 하면, t의 소멸자가 명시적으로 호출된다. 하지만 이 경우 소멸자가 2번 실행되게 된다. 그러면 이 프로그램에서는 메모리 샘(leak)이 발생하게 된다.

## 6. 명시적(explict) 생성자 호출

explict : 함축적 변환이 일어나지 않게 생성자를 선언하기 위해 사용한다.

함축적 변환(implicit conversion) : 생성자의 파라미터가 1개인 경우 대입문을 사용하여 생성자를 호출하는 것이 가능하다. 컴파일러에서 생성자로 호출하기 때문에 가능하다.

## 7. 객체 배열(object array)에서 생성자 호출

객체 배열: 클래스 형의 배열

```cpp
만약, CTest의 객체를 5개 만들려고 선언 하려면

CTest a[5];
```

디폴트 생성자 O : error가 아니다

디폴트 생성자 X : 반드시 생성자의 파라미터가 명시되어야 한다.

만약, 디폴트 생성자 X면, 어떻게 해야 하는가????????

```cpp
CTest a[5] = {1,2,3,4,5};
```

일반 배열 초기화 같지만, 이것은 생성자 호출의 함축적 변환이다.

생성자가 파라미터를 2개 이상 받는 경우,

```cpp
CTest a[5] = {(1,10),(2,20),(3,30),(4,40),(5,50)};
```

그럴듯하게 보이지만, error다. 객체 배열에서 생성자 호출은 클래스의 이름을 포함하여야 한다.

```cpp
CTest a[5] = {CTest(1,10),CTest(2,20),CTest(3,30),CTest(4,40),CTest(5,50)};
```

파라미터가 1개인 경우에도 사용할 수 있다.

```cpp
CTest a[5] = {CTest(1),CTest(2),CTest(3),CTest(4),CTest(5)};
```

이렇게 하면된다. 하지만 권장하는 방법은 아니다. 만약 배열의 크기가 100이라면, 이런것을 100개 만들어야 한다.

→ 디폴트 생성자를 만드는 것이 좋다.

동적 배열은 어떤가???

→ 동적 배열은 디폴트 생성자가 반드시 있어야 할 수 있다.

## 8. 복사 생성자(copy constructor)

함수의 파라미터로 객체가 전달하는 경우가 있다. 이 객체가 파라미터로 전달될 때는 이미 만들어진 객체의 내용이 스택에 복사되어 전달될 것이다. 이 때 생성자는 호출 되는가

```cpp
#define _CRT_SECURE_NO_WARNINGS

#include <stdio.h>
#include <string.h>
#include <conio.h>

class CStr
{
public:
	CStr(char* s)
	{
		str = new char[strlen(s) + 1];
		strcpy(str, s);
		printf("constructor");
	}
	~CStr()
	{
		delete[] str;
		printf("destructor\n");
	}
	void Print() { printf("%s\n", str); }
private:
	char* str;
};

void f(CStr s) {
	s.Print();
}

int main() {
	char hello[10] = "hello";
	CStr s(hello);

	f(s);

	return 0;
}
```

→ 결론부터 말하면, **생성자는 호출되지 않는다. 하지만 소멸자는 호출된다.**

생성자 1번 실행에 소멸자 2번 실행 → 이미 소멸한 메모리 공간에 또 한번 소멸자가 실행되기 때문에 error가 발생한다.

### 해결책

1. 객체를 전달하지 않고 포인터를 전달한다.
2. 복사 생성자(copy constructor)를 만든다.

### 1번 해결책

```cpp
....
void f(CStr& s) {
	s.Print();
}
....
```

**생성자에서 동적으로 메모리를 할당하는 객체를 함수의 파라미터로 전달 할 때는 항상 래퍼런스를 전달한다.**

### 2번 해결책

더 일반적인 해결책이다. 복사 생성자를 만드는 것이다.

**이미 만들어진 객체가 다른 곳으로 복사될 때 호출되는 생성자를 의미한다.** 객체가 복사될 때는

1. 함수의 파라미터로 객체를 전달
2. 선언에서 초기화 문장
3. 오버로드된 연산자 혹은 함수가 객체를 리턴하는 경우
4. 임시 객체가 복사되는 경우

복사 생성자 작성하는 법

- 반드시 자신의 상수 클래스 형의 래퍼런스를 파라미터로 받는다.

```cpp
CStr (const CStr &s)
{
	....
}
```

위 문장은 s가 변경되지 못함과, s의 레퍼런스가 전달됨을 지시한다. 복사 생성자는 항상 첫 번째 파라미터만이 명시 되어야 한다.

이렇게 하면,  객체가 다른 곳으로 복사될 때, 이 복사 생성자가 호출된다. 그리고 소멸자가 실행 될 때, 복사 생성자를 생성한 객체는 그 생성자를 소멸한다.

## 9. 복사 생성자가 호출되는 경우

1. 함수의 파라미터로 객체를 전달
2. 함수가 객체를 리턴할 때
3. 객체의 선언에 사용된 대입연산자
4. 임시 객체의 복사
5. 디폴트 파라미터에 명시된 생성자

```cpp
CStr s("hello");
CStr t = s; // 복사 생성자를 호출

CStr s("hello");
CStr t;
t = s; //복사 생성자 호출 X
```

아래것은 비트 단위 복사(bitwise copy)가 이루어진다. s의 내용을 1bit도 남김없이 그대로 t로 복사된다.

어떤 문제가 발생하는 가????

여러가지 있지만, 비트 단위로 복사가 이루어지기 때문에, t가 원래 참조하고 있던 값이 s의 값으로 복사가 이루어져서, t의 원래 참조값이 사라질 수 있다.

이런 문제를 해결하기 위해서는 완벽한 규칙이 필요하다

- **생성자에서 동적으로 메모리를 할당하는 경우, 반드시 복사 생성자를 만들라. 또한, 반드시 대입 연산자(=)를 오버로딩하여, 레퍼런스를 리턴하도록 해야한다.**

## 10. 리턴형의 (void)형 변환

```cpp
#include <stdio.h>

int f(int x) {
	printf("%d\n", x * x);
	return x * x;
}

void main() {
	(void)f(3);
}
```

f(3)과 (void)f(3)의 차이

→ (void)는 f()에서 main()으로 돌려주기 위한 리턴값의 복사를 위한 코드가 생성되지 않는다. 이것은 최적화에 도움이 된다. (의도적으로, 그리고 이것은 실제 확인된 사례가 아니다.)

## 11. 베이스 클래스와 멤버 객체의 생성자 호출

베이스 클래스의 생성자 호출을 위해  ‘ : ’을 사용한다.

생성자가 호출되는 순서는 왼쪽에서 오른쪽으로 호출된다. 그리고 자식 클래스의 생성자가 가장 마지막에 호출된다.

```cpp
B::B() : A(value1), C(value2) {
...
}
// 생성자 호출 순서: 1. A() 2. C() 3. B()
```

콜론 ‘ : ‘ 다른 용도로도 사용된다.

- 상수 멤버 의 초기화
- 참조의 초기화
- 객체 멤버의 초기화
- 베이스 클래스의 생성자 호출
- 일반 데이터 멤버의 초기화(권장 X)

## 12. 생성자와 소멸자의 접근 권한

생성자와 소멸자는 대부분 public으로 공개된다.

protected경우 베이스 클래스로만 사용될 것을 문서화한다.

소멸자와 생성자는 private 권한을 가질 수 없다.

→ 멤버 함수이기 때문에, 외부에서 호출할 수 있기 위해서는 public 접근 권한을 가져야 하기 때문이다.

하지만, 고의로 private로 선언하는 경우가 3가지 있다.

- 특정한 함수에서만 객체를 생성할 필요가 있는 경우
- 특정한 클래스에서만 객체를 생성할 필요가 있는 경우
- 디폴트 생성자 호출을 막고 싶은 경우

## 13. static 객체 생성과 소멸

### static

static 선언은 실행 파일이 로드될 때 한 번만 실행되어 초기화된 후, static을 참조하는 모든 문장은 로드될 때 결정된 주소를 참조하는 것이다.

→ 전역 변수와 같은 생명 주기를 가졌다.

하지만 static이 그 범위를 제한하는 역할을 하는 것이고, 이러한 범위 제한의 검사는 컴파일 시간에 결정된다.

```cpp
#include <stdio.h>

class Ctest
{
public:
	Ctest(int i) {
		m_i = i;
		printf("constructor\n");
	}

	~Ctest() {
		printf("destructor\n");
	}

	void Print() {
		printf("i=%d\n", m_i);
	}

private:
	int m_i;
};

void f(int j) {
	static Ctest t(j);

	t.Print();
}

void main() {
	printf("begin main()\n");
	f(1);
	f(2);
	printf("end main()\n");
}

// 실행 결과
/*
begin main()
constructor
i=1
i=1
end main()
destructor
/*
```
