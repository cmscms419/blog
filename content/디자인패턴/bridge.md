# bridge

# bridge는 무엇인가?

bridge 패턴은 서로 관련된 클래스들의 집합을 두 개의 별개의 계층구조(추상화 및 구현)로 나눈 후 각각 독립적으로 개발할 수 있도록 하는 구조 디자인 패턴입니다.

# bridge 패턴을 왜 사용하는 가?

예를 들어

`원`과 `직사각형` 이라는 한 쌍의 자식 클래스들을 가진 `모양` 클래스

`빨간색`과 `파란색`이라는 한 쌍의 자식 클래스들을 가진 `색` 클래스

처음에 `모양` 클래스가 있고, 이 클래스에 `색` 클래스가 추가 되었다고 하면

자식 클래스의 숫자는 총 4가지가 나와야 합니다.

![[/problem-ko.png]]

새로운 유형을 추가하려고 하면, 계층 구조는 기하급수적으로 늘어납니다.

자식 클래스의 숫자가 너무 많아집니다.

## bridge 패턴을 사용한다면?

차원 중 하나를 별도의 클래스 계층 구조로 추출하여 원래 클래스들이 한 클래스 내에서 모든 상태와 행동들을 갖는 대신 새 계층구조의 객체를 참조하도록 한다.

→ 1개의 클래스를 2개로 나누어서, 다른 하나를 참조해서 사용한다.

![[/solution-ko.png]]

추상화 객체는 앱의 드러나는 모습을 제어하고, 연결된 구현 객체에 실제 작업들을 실행시킵니다.

→ `모양` 클래스로 제어, `원, 직사각형` 클래스로 실제 작업

→ 서로 다른 구현들은 공통 인터페이스를 따르는 한 상호 호환이 가능하며, 플랫폼에 동시에 작동 가능함

→ `원, 직사각형` 클래스를 건드리지 않고, `모양` 클래스들을 변경할 수 있습니다.

→ 다른 OS에 대한 지원을 하려면 구현 계층구조 내에 자식 클래스를 생성해주면 됩니다.

## 엔진에서 bridge 패턴이 사용된 이유

bridge 패턴이 사용된, 객체는

```cpp
FontBridge
SurfaceBridge
TextureBridge
```

이 있습니다.

이 클래스들의 특징

- 엔진에서의 관리 방식과 시스템에서 지원하는 방식을 연결합니다.
- 시스템의 데이터를 가져와서, gles_xxxx 코드에 적용 시켜줍니다.
    - 예)
    
    ![Untitled](attachments/Untitled%201.png)
    
- factory + adapter이 연관되어 있습니다.

데이터를 가져오는 객체와 그 데이터를 기반으로 API를 통해 만드는 객체를 합쳐서 만들면, 나중에 확장하기 어려워 집니다.

이 문제를 해결하기 위해서, 두 개의 객체로 나눠서 만드는 것입니다.

Core에서 추상화를 통해서

API를 통해 만드는 객체는 오직 bridge를 통해서 온 데이터를 토대로 객체를 만듭니다.

그리고 bridge는 오직 시스템에 있는 데이터를 어떻게 하면 API가 인식할 수 있게 만들어서 전달해 주는 것만 생각하면 됩니다.
