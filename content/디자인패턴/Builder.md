# Builder

# 이 패턴에 대한 설명(의도)

복잡한 객체들을 단계별로 생성할 수 있도록 하는 생성 디자인 패턴입니다.

이 패턴을 사용하면 제작 코드를 사용하여 객체의 다양한 유형들과 표현을 제작할 수 있습니다.

# 이 패턴이 필요한 이유

많은 필드 + 중첩된 객체들을 단계별로 초기화 해야하는 객체를 초기화 하기 힘듭니다.

이 초기화 코드 보통 생성자 내부에 있습니다.

더 최악은 클라이언트 코드 전체에 흩어져 있을 수 있습니다.

### 예를 들어)

`House` 클래스가 있는데,

이 클래스는 4개 벽, 바닥, 문, 지붕을 만듭니다.

하지만, 추가적으로 창문, 뒤뜰. 하수도가 있는 집을 원한다면???

1. 방법

기초 `House` 클래스를 확장하고 매개변수의 모든 조합을 포함한 자식 클래스들의 집합을 만들기

→ 안됨 → 상당한 수의 자식 클래스를 만들게 되고, 새 매개변수가 추가될 때마다 계층 구조가 복잡해집니다.

1. 방법

기초 `House` 클래스에 `House` 객체를 제어하는 모든 매개변수를 포함한 생성자를 만드는 것

→ 자식 클래스 필요 X

하지만 안됨  → 모든 매개변수가 항상 필요한 것이 아니다.

생성자 호출 코드가 매우 못생겨짐

# 여기서 Builder 패턴

복잡한 객체들을 단계별로 생성할 수 있도록 합니다.

빌더가 객체를 생성되는 동안 다른 객체들이 제품에 access하는 것을 허용하지 않습니다.

예를 들어)

![Untitled](attachments/Untitled_17.png)

벽, 문, 창문, 지붕 등의 객체를 생성하는 메소드

위 메소드를 생성하고 싶으면 위 단계들을 빌더 객체에 실행해 주면 됩니다.

장점

필요한 메소드만 호출하면 됩니다.

공통 인터페이스를 사용하여, 서로 다른 빌더를 구현하여, 생성되는 벽과 문의 개수는 같지만, 완성된 형태를 다르게 할 수 있습니다.

### 디렉터

디렉터 클래스 : 빌더를 통해 제작 단계들의 실행 순서를 정의합니다.

필수는 아닙니다.

# 왜 엔진 초기화 할 때, build 패턴을 사용하는 가??

엔진에 넘겨줄 정보가 gles, gl, directx10 일 수 도 있다.

그 모든 그래픽 엔진에 대해서 클래스를 구현하기에는 유지보수적 관점에서 좋지 못합니다.

이것을 해결하기 위해, 빌더 패턴을 사용

빌더 패턴을 사용하면, 엔진마다 필요한 정보를 넘겨줄 빌더를 구현할 수 있습니다.
